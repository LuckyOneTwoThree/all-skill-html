# Pipeline 20: 后端架构审查自动执行

## 核心原则

1. **可量化检查**：所有检查项有明确通过/不通过标准
2. **风险分级**：P0阻塞发布 / P1本迭代修复 / P2下迭代优化
3. **修复可执行**：每个问题附带代码级修复建议
4. **防退化**：审查规则写入CI，不通过不合并

## 交互模式

🤖 AI自动执行

## 输入

| 输入项 | 类型 | 必填 | 来源 | 说明 |
|--------|------|------|------|------|
| 服务设计 | JSON | 是 | output/backend-architecture/service-design | 服务架构和通信方案 |
| API契约 | YAML/JSON | 是 | output/backend-api-design/api-contract/openapi.yaml | 接口定义 |
| 数据模型 | JSON | 是 | output/backend-data-architecture/data-model/er_model.json | 数据库设计 |
| 缓存策略 | JSON | ○ | output/backend-data-architecture/cache-strategy | 缓存方案 |

## 执行步骤

### Step 1: 性能审查

检查后端性能相关设计：

| 检查项 | 通过标准 | 级别 |
|--------|---------|------|
| 接口响应时间 | P99<500ms（读）/ <1s（写） | P0 |
| 数据库查询 | 无N+1查询，单查询<100ms | P0 |
| 连接池配置 | 连接池大小≥并发数/2 | P1 |
| 分页查询 | 列表接口100%有分页 | P0 |
| 慢查询防护 | 查询超时设置≤5s | P1 |
| 异步处理 | 耗时>1s的操作异步化 | P1 |
| 批量操作 | 批量接口单次≤100条 | P1 |
| 缓存命中率 | 热点数据缓存命中率≥80% | P2 |

### Step 2: 安全审查

检查后端安全相关设计：

| 检查项 | 通过标准 | 级别 |
|--------|---------|------|
| SQL注入防护 | 100%参数化查询 | P0 |
| 认证鉴权 | L2+接口100%有鉴权 | P0 |
| 敏感数据加密 | 密码bcrypt/手机号AES加密 | P0 |
| 日志脱敏 | 日志中无明文敏感数据 | P0 |
| 限流防护 | 100%接口有限流策略 | P1 |
| CORS配置 | Origin白名单非* | P1 |
| 依赖安全 | 无已知高危CVE依赖 | P0 |
| 密钥管理 | 密钥不在代码中硬编码 | P0 |

### Step 3: 可维护性审查

检查代码和架构可维护性：

| 检查项 | 通过标准 | 级别 |
|--------|---------|------|
| 代码分层 | Controller/Service/Repository三层分离 | P1 |
| 循环依赖 | 模块间无循环依赖 | P0 |
| 接口文档 | 100%接口有OpenAPI文档 | P1 |
| 错误处理 | 统一错误码+错误响应格式 | P1 |
| 日志规范 | 统一日志格式+链路追踪ID | P1 |
| 配置管理 | 环境配置外置，不硬编码 | P1 |
| 代码重复 | 重复代码率<5% | P2 |
| 函数复杂度 | 单函数圈复杂度≤15 | P2 |

### Step 4: 可扩展性审查

检查架构可扩展性：

| 检查项 | 通过标准 | 级别 |
|--------|---------|------|
| 无状态设计 | 服务无本地状态，可水平扩展 | P0 |
| 数据库扩展 | 有分库分表预案（数据量>500万行时） | P1 |
| 服务拆分 | 单服务代码≤50000行 | P1 |
| 消息解耦 | 跨服务写操作通过事件解耦 | P2 |
| 配置热更新 | 配置变更不重启服务 | P2 |
| 灰度发布 | 支持按比例/按用户灰度 | P2 |

### Step 5: 问题汇总与修复建议

汇总所有问题并生成修复建议：

- 按P0/P1/P2分级
- 每个问题附带：模块、问题描述、修复建议、预估工作量
- 统计通过率和问题分布
- 生成修复优先级排序

## 决策规则

| 条件 | 决策 |
|------|------|
| P0问题>0 | 阻塞发布，必须修复 |
| P0=0 且 P1≤3 | 可发布，标注已知问题 |
| P0=0 且 P1>3 | 建议修复后再发布 |
| 通过率≥90% | 标记"高质量" |
| 通过率75%-90% | 标记"合格" |
| 通过率<75% | 标记"需改进" |
| 同类问题≥3个 | 标记"系统性问题" |

## 质量检查

- [ ] 审查覆盖性能/安全/可维护性/可扩展性4个维度
- [ ] 每个检查项有明确的通过标准
- [ ] P0问题100%有修复建议
- [ ] 问题按P0/P1/P2分级
- [ ] 修复建议包含预估工作量

## 降级策略

| 缺失的上游输入 | 降级方案 | 输出影响 |
|---------------|---------|---------|
| 缓存策略缺失 | 跳过缓存命中率检查 | 无法评估缓存性能 |
| 服务设计缺失 | 跳过可扩展性审查 | 无法评估架构可扩展性 |
| API契约缺失 | 跳过接口级安全检查 | 无法评估接口安全性 |

  3. 描述后端架构和核心接口
